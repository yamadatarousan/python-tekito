# コードリーディングガイド

このドキュメントは、初見でもこのプロジェクトのコードを短時間で理解できるように、
「どこから読むか」「何を確認するか」を整理したものです。

## 1. 先に把握する全体像

このプロジェクトは、責務を分けた典型的な構成です。

- `routers`: HTTPエンドポイント。入力の受け取りとレスポンス整形。
- `services`: ユースケース。業務ルールの実行。
- `repositories`: 永続化。SQLAlchemyモデルへのアクセス。
- `domain`: DBやWebから独立した純粋ロジック（集計など）。
- `infrastructure`: DB接続やORMモデルなどの技術的な土台。
- `schemas`: API入出力のPydanticモデル。

重要な前提は「ルーターが直接DBモデルを触らない」ことです。
処理は基本的に `router -> service -> repository -> model` の向きで流れます。

## 2. 最短で理解する読み順

### 手順0: エントリポイント

1. `app/main.py`
- `create_app()` がアプリ初期化の中心です。
- `Base.metadata.create_all()` が起動時に実行されるため、テーブル作成タイミングがここで分かります。
- APIルーターとWebルーターの登録場所もここです。

### 手順1: 設定と依存注入

2. `app/core/config.py`
- 既定の `database_url`、環境変数プレフィックスを確認します。

3. `app/dependencies.py`
- 1リクエストごとのDBセッション生成とクローズ処理を確認します。

### 手順2: 入口（API / Web）

4. `app/routers/api.py`
- API仕様の入口です。
- `_build_services()` で毎リクエスト、必要サービスを組み立てています。

5. `app/routers/web.py`
- サーバーサイド描画の入口です。
- APIと同じサービスを呼び出すため、業務ルールの一貫性を追いやすい構成です。

### 手順3: ユースケース

6. `app/services/task_service.py`
- 入力の正規化（`strip()`）や存在チェックなど、実際の判断ロジックが集まっています。

7. `app/services/project_service.py`
8. `app/services/dashboard_service.py`
- プロジェクト操作とダッシュボード集計呼び出しを確認します。

### 手順4: 永続化

9. `app/repositories/project_repository.py`
10. `app/repositories/task_repository.py`
- SQLAlchemy問い合わせと、DBモデルからドメインモデルへの変換責務を確認します。

### 手順5: ドメインとDBモデル

11. `app/domain/entities.py`
- `TaskStatus` の定義がAPI/DB/Webで共通の中心概念です。

12. `app/domain/dashboard.py`
- 期限切れ判定や進捗率計算など、仕様の解釈が必要なロジックがあります。

13. `app/infrastructure/models.py`
- `projects` / `tasks` テーブル構造と関連を確認します。

## 3. 1リクエストを追うときの見方

### 例1: APIでタスク作成

`POST /api/projects/{project_id}/tasks` は次の順に進みます。

1. `app/routers/api.py#create_task`
2. `app/services/task_service.py#create_task`
3. `app/repositories/task_repository.py#create`
4. `app/infrastructure/models.py#TaskModel` に保存
5. `app/schemas/api.py#TaskResponse.from_domain` で返却整形

確認ポイント:
- プロジェクト存在チェックは `TaskService` で実施。
- タイトル空文字チェックも `TaskService` で実施。
- DBの型制約ではなく、サービス層でもバリデーションしている。

### 例2: Webでタスク状態を進める

`POST /tasks/{task_id}/cycle` は次の順に進みます。

1. `app/routers/web.py#cycle_task_status`
2. `app/services/task_service.py#cycle_status`
3. `app/services/task_service.py#update_status`
4. `app/repositories/task_repository.py#update_status`
5. リダイレクトで詳細画面へ復帰

確認ポイント:
- 状態遷移の仕様（todo -> in_progress -> done -> todo）は `TaskService` にあります。
- UI側に状態遷移ロジックは置いていません。

## 4. テストを仕様書として読む順番

1. `tests/test_dashboard_service.py`
- 集計仕様（期限切れ条件、0件時の進捗率）を先に把握できます。

2. `tests/test_api_and_web.py`
- APIの基本ユースケースと、Web画面の最低限の動作保証を把握できます。

意図コメントが丁寧に書かれているため、コード変更前に先に読むと仕様誤読を減らせます。

## 5. 変更時の逆引き

### APIの項目追加・変更

- 入口: `app/routers/api.py`
- スキーマ: `app/schemas/api.py`
- 業務ルール: `app/services/*.py`
- 永続化: `app/repositories/*.py`, `app/infrastructure/models.py`
- テスト: `tests/test_api_and_web.py`

### 集計ルール変更（期限切れや進捗率）

- 仕様実装: `app/domain/dashboard.py`
- 呼び出し: `app/services/dashboard_service.py`
- 表示: `app/templates/index.html`
- テスト: `tests/test_dashboard_service.py`

### 画面フォーム変更

- ルーティング/フォーム受け取り: `app/routers/web.py`
- テンプレート: `app/templates/*.html`
- スタイル: `app/static/style.css`
- 回帰確認: `tests/test_api_and_web.py`

## 6. ハマりどころ

- `create_all()` は起動時に実行されるため、学習では便利ですが本番向け運用とは分けて考える必要があります。
- `web.py` の日付入力は空文字で来るため、`date.fromisoformat()` 前の空判定が必須です。
- `TaskStatus` は文字列Enumです。DB文字列と一致しない値を使うと変換時に失敗します。
- 一部バリデーションはPydantic（`schemas`）とServiceの両方にあります。責務重複に見えても、入口別（API/Web）の安全性を担保する意図があります。
